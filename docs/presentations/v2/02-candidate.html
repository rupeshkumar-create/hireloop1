Match scoring · formula

How overall_score is calculated

Core math lives in MatchingEngine._assemble_score (services/matching.py).

overall = Σ (w_i × dim_i) / Σ w_included # then × gates (role · domain · title · seniority) w_skills = 0.40 w_profile = 0.30 w_experience = 0.15 w_location = 0.10 w_ctc = 0.05 # missing exp/CTC dims → weights renormed

Match scoring · dimensions

Where each dim comes from

Skills (0.40)
Lexical: 0.85×coverage + 0.15×jaccard on canonical skills. Then semantic lift from cosine(ce.skills_embedding, je.skills_embedding) — embeddings only raise the lexical base (lift α=0.40).
Profile (0.30)
Cosine(ce.profile_embedding, je.jd_embedding) with calibrated lift (α=0.50).
Experience (0.15)
Only if job seniority set — in-band 1.0, under/over linear decay.
Location (0.10)
Remote+open→1.0 · same city 1.0 · country/global 0.9 · state 0.7 · else 0.2
CTC (0.05)
Overlap vs job band; missing data → 0.5

Match scoring · gates & floors

After the weighted mean

Gates (multipliers)

Role-fit gate · domain_fit (e.g. hospitality↔tech harsh) · generic-title penalty (×0.35) · seniority gap (1.0 → 0.4)

Quality floors

Persist ≥ 0.35 (path-aligned ≥0.18) · feed ≥ 0.38 · weak role signal dropped. Failed persist deletes old match_scores row.

Written → public.match_scores overall_score, skills_score, experience_score, location_score, ctc_score, bias_audit, explanation*, title_score, ranking_model_version=v2_feature_rerank…

Ranking ≠ scoring

RRF is for ordering the feed

overall_score is computed first. Reciprocal Rank Fusion blends lists for presentation / Aarya search — it does not redefine the weight formula.

RRF: score += weight / (k + rank + 1) k = 60 Feed first screen: fuse (overall, skills) → MMR λ=0.72 · company cap 2 Aarya job_search: multi-pool recall → RRF + lift 0.08·overall + 0.04·skills + 0.02·exp Feature rerank (search): 0.30 title + 0.20 req_skills + 0.15 resp + …
